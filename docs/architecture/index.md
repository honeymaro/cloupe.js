# Architecture

Describes the internal structure of cloupe.js and the .cloupe file format.

## .cloupe File Format

.cloupe files are a binary file format used by 10x Genomics Loupe Browser. They are generated by the Cell Ranger pipeline and store single-cell genomics data.

### File Layout

```
┌─────────────────────────────────┐
│ Header (4096 bytes, JSON)       │  ← Version, index block location
├─────────────────────────────────┤
│ Index Block (JSON, gzip)        │  ← Offsets for all data sections
├─────────────────────────────────┤
│ Data Blocks                     │
│  ├─ Runs (metadata)             │
│  ├─ Metrics (quality metrics)   │
│  ├─ Projections (UMAP, t-SNE)   │
│  ├─ Matrices (CSC sparse matrix)│
│  │   ├─ Barcodes                │
│  │   ├─ Features (Genes)        │
│  │   └─ Expression data         │
│  ├─ Clusterings                 │
│  └─ CellTracks (annotations)    │
└─────────────────────────────────┘
```

### Header Structure (4096 bytes)

The first 4096 bytes contain a JSON-formatted header:

```json
{
  "version": "0.7.0",
  "headerSize": 4096,
  "blockOffset": 4096,
  "indexBlock": {
    "Start": 4096,
    "End": 12345,
    "CompressionType": 1
  },
  "nextHeaderOffset": null,
  "timestamp": "2024-01-01T00:00:00Z"
}
```

### Index Block

The index block is gzip-compressed JSON containing location information for all data blocks:

```json
{
  "Matrices": [
    {
      "Name": "Gene Expression",
      "GeneCount": 32738,
      "CellCount": 8414,
      "Barcodes": { "Start": 12345, "End": 23456, "ArraySize": 8414, "ArrayWidth": 18 },
      "Genes": { "Start": 23456, "End": 34567, "ArraySize": 32738, "ArrayWidth": 15 },
      "CSCValues": { "Start": 34567, "End": 45678, "CompressionType": 2 },
      "CSCIndices": { "Start": 45678, "End": 56789, "CompressionType": 2 },
      "CSCPointers": { "Start": 56789, "End": 67890 }
    }
  ],
  "Projections": [
    {
      "Name": "UMAP",
      "Dims": [2, 8414],
      "Matrix": { "Start": 67890, "End": 78901, "CompressionType": 1 }
    }
  ],
  "CellTracks": [
    {
      "Name": "Cluster",
      "NumCategories": 10,
      "Categories": { "Start": 78901, "End": 89012 },
      "Values": { "Start": 89012, "End": 100123 }
    }
  ]
}
```

### Data Types

| Data                        | Byte Format        | Description     |
| --------------------------- | ------------------ | --------------- |
| Strings (barcodes/features) | UTF-8, NULL padded | Fixed width     |
| Projection coordinates      | double (8B)        | Float64         |
| UMI counts                  | uint64 (8B)        | Unsigned 64-bit |
| CellTrack values            | int16 (2B)         | Signed 16-bit   |
| CSC indices/pointers        | uint64 (8B)        | Unsigned 64-bit |

### Compression

Three compression methods are used:

| CompressionType | Description                                               |
| --------------- | --------------------------------------------------------- |
| 0               | No compression                                            |
| 1               | Standard gzip (magic bytes: 0x1F 0x8B)                    |
| 2               | Block-indexed compression (index of multiple gzip blocks) |

#### Block-Indexed Compression (CompressionType=2)

Large data uses block-indexed compression:

```
┌─────────────────────────────────┐
│ Index Block                     │
│  [blockSize, offset0, offset1,  │
│   offset2, ..., offsetN]        │
├─────────────────────────────────┤
│ Compressed Block 0              │
├─────────────────────────────────┤
│ Compressed Block 1              │
├─────────────────────────────────┤
│ ...                             │
├─────────────────────────────────┤
│ Compressed Block N              │
└─────────────────────────────────┘
```

- Data is split into ~65KB chunks
- Each block is independently gzip compressed
- Index block stores offsets for each compressed block
- Enables random access (decompress only needed blocks)

## CSC Matrix Format

Expression matrices are stored in CSC (Compressed Sparse Column) format:

### CSC Structure

```
Matrix (3 genes x 4 cells):
         Cell0 Cell1 Cell2 Cell3
Gene0  [   1     0     2     0  ]
Gene1  [   0     0     3     0  ]
Gene2  [   4     5     0     6  ]

CSC format:
  data:    [1, 4, 5, 2, 3, 6]     (non-zero values in column-major order)
  indices: [0, 2, 2, 0, 1, 2]     (row index for each value)
  indptr:  [0, 2, 3, 5, 6]        (start position for each column)
```

### CSC → CSR Conversion

cloupe.js internally converts to CSR (Compressed Sparse Row) format to optimize row access:

```typescript
// CSC (column-based): Fast access to all gene expressions for a specific cell
const cellExpression = csc.getColumn(cellIndex);

// CSR (row-based): Fast access to all cell expressions for a specific gene
const geneExpression = csr.getRow(geneIndex);
```

## Design Principles

### 1. Lazy Loading

Only the header and index are parsed when opening a file. Actual data is read only when requested.

```typescript
// Only header/index parsed
const reader = await CloupeReader.open(file);

// Barcode data is read at this point
const barcodes = await reader.getBarcodes();
```

### 2. Random Access

Uses the File API's `slice()` to read only the required portion of the file:

```typescript
// Internal implementation
const chunk = file.slice(start, end);
const buffer = await chunk.arrayBuffer();
```

### 3. Caching

Frequently used data is cached to prevent repeated reads:

```typescript
// First call: reads from file
const barcodes1 = await reader.getBarcodes();

// Second call: returns from cache
const barcodes2 = await reader.getBarcodes();

// Clear cache
reader.clearCache();
```

### 4. Type Safety

All public APIs are typed with TypeScript:

```typescript
interface Feature {
  index: number;
  id: string;
  name: string;
  type?: string;
}

async getFeatures(): Promise<Feature[]>
```

## Browser Limitations

### Memory

- Be cautious of memory usage when loading full matrix
- Web Workers recommended for large files
- Use `clearCache()` actively

### File API

- `slice()` performance varies by browser
- Chrome/Edge are the fastest

### SharedArrayBuffer

- Requires CORS headers:
  - `Cross-Origin-Opener-Policy: same-origin`
  - `Cross-Origin-Embedder-Policy: require-corp`
